## 1. Project Overview

**Gharka Chef** is an AI-powered Indian cooking assistant web application.
Users interact via a chat interface and receive:

- Step-by-step recipe instructions (generated by LLM)
- Ingredient images (AI-generated, cached)
- Nutrition information (IFCT-based, Indian food composition)
- Chef profiles and video content
- *(Phase 2)* Infographic recipe cards
- *(Phase 3)* Auto-generated short cooking videos & social posts

**Target Audience:** Indian home cooks and food enthusiasts.
**Platform:** Web-first, mobile-friendly. No WhatsApp/app download required.

```sql
ingredient_alias    → id, alias TEXT (UNIQUE), canonical_name, created_at
recipe_search_log   → id, recipe_key, searched_at, user_id → users
```

### Chat

```sql
chat_session        → id, user_id → users, created_at, updated_at
chat_message        → id, chat_id → chat_session, role (user|assistant),

chat_content_block  → id, message_id → chat_message, block_type (text|recipe|video|ad|cta),
```

### Engagement

```sql

bookmarks           → id, user_id → users, recipe_id → recipe_master (UNIQUE pair)
step_reaction_user  → id, user_id, recipe_id, step_index, reaction (like|dislike) (UNIQUE triplet)

recipe_reaction_user → id, user_id, recipe_id, reaction (like|dislike), chat_id (UNIQUE user+recipe)
```

### Videos

```sql
videos              → id, user_id → users, recipe_id → recipe_master, url, title,
```

## 8. LLM / Recipe Generation Pipeline

```
User message
    ↓
POST /chat (api/routes.py)
    ↓
1. Auth check (Bearer token)
    ↓
2. Extract recipe name / intent from message
    ↓
3. LAYER 1: DB cache check (recipe_key lookup)
    ↓ miss
4. LAYER 2: Vector similarity search (RecipeRetriever)
   → Threshold: 0.85 similarity required
   → Also checks intent keywords (high_protein, low_carb, vegan, etc.)
    ↓ miss
5. Call LLM (configured provider):
   → OpenAI:  client.responses.create (gpt-4o-mini)
   → Groq:    OpenAI-compatible client (llama-3.1-8b-instant)
   → Ollama:  Python client or CLI subprocess (llama3.1:latest)
    ↓
6. JSON extraction & repair:
   → _extract_json()       — strip markdown fences, find JSON object
   → _clean_json_for_parse() — fix LLM-specific corruption
   → json.loads()
    ↓
7. Field normalization:
   → _coerce_single_recipe() — maps any LLM field names → canonical schema
   → _validate_and_coerce()  — Pydantic validation + graceful fallback
    ↓
8. Save to DB:
   → recipe_master (recipe_key, title, steps_json, recipe_json)
   → ingredient_master (per ingredient, upsert)
   → recipe_ingredient_map (recipe ↔ ingredient links)
    ↓
9. Return structured JSON to frontend
```

## 9. Authentication Flow

### Email + Password (direct login)
```
POST /auth/register { full_name, email, password }
  → Saves user with password_hash (SHA-256 + salt), is_verified=true
POST /auth/login { email, password }
  → Verifies hash → creates token in user_token → returns { token, user_id }
```

### Email OTP Flow
```
POST /auth/register { full_name, email }  (no password)
  → Creates user + OTP → emails OTP (or logs in dev mode)
POST /auth/verify-otp { email, code }
  → Verifies OTP → marks user verified → creates token → returns { token, user_id }
```

### Using Auth in Frontend
- Store `token` in localStorage
- Send as `Authorization: Bearer <token>` header
- `GET /auth/me` returns full user profile

### Dev Mode Helper
```
POST /auth/dev-setup?email=test@local
  → Creates test user + sample recipe + returns token (only when DEV_RETURN_OTP=true)
```

### Registration Fix Details (Bug #3)
- **Root cause**: `email-validator` 2.x rejects emails without a proper domain (e.g. `user@localhost`, `test@local`). Frontend silently discarded the 422 error body so users saw a generic "Registration failed." with no explanation.
- **Fix applied**: Added `_getApiError()` helper that reads Pydantic error detail from the API response. Added `_isValidEmail()` client-side regex check. Registration and login errors now show the specific reason. Password length minimum (6 chars) enforced on frontend. Whitespace trimmed from email/name before sending.

## 13. Development Rules & Conventions

1. **Test on localhost first** — All changes are tested locally before any deployment.
2. **No breaking schema changes** — `schema.sql` uses `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for safe idempotent migrations.
3. **Cache-first architecture** — AI/LLM is called ONLY when data is not in DB or vector cache.
4. **DB access pattern** — Always use `get_connection()` as context manager with `dict_row`. No ORM.
5. **LLM is provider-agnostic** — New LLM providers can be added by extending `generate_recipe()` in `recipe_service.py`.
6. **Frontend API base URL** — `http://localhost:8005` (`NEXT_PUBLIC_API_URL` env var; default in all pages if not set).
7. **Auth** — All protected endpoints expect `Authorization: Bearer <token>` header.
8. **Admin check** — Use `_ensure_admin(user_id)` in `admin.py` before admin operations.
9. **Media files** — Stored in `backend/media/`, served at `/media/*` via FastAPI static files.
10. **This document** — Update this file whenever significant changes are made to the project.

Manages video playback state across the page.

| Item | Detail |
|------|--------|

| Use | Video components read `playingId` to pause others when a new one plays |
