## 1. Project Overview

**Gharka Chef** is an AI-powered Indian cooking assistant web application.
Users interact via a chat interface and receive:

- Step-by-step recipe instructions (generated by LLM)
- Ingredient images (AI-generated, cached)
- Nutrition information (IFCT-based, Indian food composition)
- Chef profiles and video content
- *(Phase 2)* Infographic recipe cards
- *(Phase 3)* Auto-generated short cooking videos & social posts

**Target Audience:** Indian home cooks and food enthusiasts.
**Platform:** Web-first, mobile-friendly. No WhatsApp/app download required.

## 8. LLM / Recipe Generation Pipeline

```
User message
    в†“
POST /chat (api/routes.py)
    в†“
1. Auth check (Bearer token)
    в†“
2. Extract recipe name / intent from message
    в†“
3. LAYER 1: DB cache check (recipe_key lookup)
    в†“ miss
4. LAYER 2: Vector similarity search (RecipeRetriever)
   в†’ Threshold: 0.85 similarity required
   в†’ Also checks intent keywords (high_protein, low_carb, vegan, etc.)
    в†“ miss
5. Call LLM (configured provider):
   в†’ OpenAI:  client.responses.create (gpt-4o-mini)
   в†’ Groq:    OpenAI-compatible client (llama-3.1-8b-instant)
   в†’ Ollama:  Python client or CLI subprocess (llama3.1:latest)
    в†“
6. JSON extraction & repair:
   в†’ _extract_json()       вЂ” strip markdown fences, find JSON object
   в†’ _clean_json_for_parse() вЂ” fix LLM-specific corruption
   в†’ json.loads()
    в†“
7. Field normalization:
   в†’ _coerce_single_recipe() вЂ” maps any LLM field names в†’ canonical schema
   в†’ _validate_and_coerce()  вЂ” Pydantic validation + graceful fallback
    в†“
8. Save to DB:
   в†’ recipe_master (recipe_key, title, steps_json, recipe_json)
   в†’ ingredient_master (per ingredient, upsert)
   в†’ recipe_ingredient_map (recipe в†” ingredient links)
    в†“
9. Return structured JSON to frontend
```

## 9. Authentication Flow

### Email + Password (direct login)
```
POST /auth/register { full_name, email, password }
  в†’ Saves user with password_hash (SHA-256 + salt), is_verified=true
POST /auth/login { email, password }
  в†’ Verifies hash в†’ creates token in user_token в†’ returns { token, user_id }
```

### Email OTP Flow
```
POST /auth/register { full_name, email }  (no password)
  в†’ Creates user + OTP в†’ emails OTP (or logs in dev mode)
POST /auth/verify-otp { email, code }
  в†’ Verifies OTP в†’ marks user verified в†’ creates token в†’ returns { token, user_id }
```

### Using Auth in Frontend

## 13. Development Rules & Conventions

1. **Test on localhost first** вЂ” All changes are tested locally before any deployment.
2. **No breaking schema changes** вЂ” `schema.sql` uses `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for safe idempotent migrations.
3. **Cache-first architecture** вЂ” AI/LLM is called ONLY when data is not in DB or vector cache.
4. **DB access pattern** вЂ” Always use `get_connection()` as context manager with `dict_row`. No ORM.
5. **LLM is provider-agnostic** вЂ” New LLM providers can be added by extending `generate_recipe()` in `recipe_service.py`.
6. **Frontend API base URL** вЂ” `http://localhost:8005` (`NEXT_PUBLIC_API_URL` env var; default in all pages if not set).
7. **Auth** вЂ” All protected endpoints expect `Authorization: Bearer <token>` header.
8. **Admin check** вЂ” Use `_ensure_admin(user_id)` in `admin.py` before admin operations.
9. **Media files** вЂ” Stored in `backend/media/`, served at `/media/*` via FastAPI static files.
10. **This document** вЂ” Update this file whenever significant changes are made to the project.
